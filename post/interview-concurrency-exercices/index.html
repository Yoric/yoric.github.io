Interview Concurrency Exercices

@David Teller · Oct 11, 2020 · 3 min read

I’ve recently gone through a number of job interviews. As a warmup, I’ve attempted to find exercices in concurrency and I was a bit disappointed by what I’ve found. So here are a few, from the top of my head. I’ll add more as I think of them.

General

  • What kind of concurrency primitives does offer or recommend (either in the language or the standard library)? Tell me about why these primitives and not others? What are the tradeoffs of these primitives?

  • When would you pick fibers (aka coroutines aka async functions) / threads / processes? What are the tradeoffs of each paradigm?

  • What are atomics? Why do load and store atomic operations take as parameter an ordering? When would you use atomics rather than mutexes?

Threads

  • Consider an array-like structure of length N, where N is very large. The array contains unordered values. Write some code to search for a value V in this structure with T threads. What are the tradeoffs? How will you optimize T for the fastest search?

  • Design and implement a thread pool. How could you tweak it to work well with long, non-blocking, I/O.

  • Consider a function foo. This function is unreliable, insofar as it sometimes freezes. Design a mechanism to call foo and retry as often as needed to make sure that the call has succeeded. You will need to make additional hypotheses on foo.

  • How would you go about implementing a thread-safe memory allocator?

  • How would you go about implementing thread-safe reference-counting?

  • Consider a multi-threaded application with a main thread and any number of worker threads. Consider a data structure Foo. This data structure is reference-counted but the destructor is designed in such a way that it can only be safely executed in the main thread. Consider a case in which worker threads need to manipulate instances of Foo. How can you organize or refactor your code to ensure that the destructor is only ever executed on the main thread?

  • Consider a function foo. This function is slow and blocking, so we wish to write a wrapper foo_coalesced that coalesces calls to foo: if we call foo_coalesced and nobody is currently calling foo_coalesced, call foo and return the result but if we call foo_coalesced and another call to foo_coalesced is in progress, merge the calls (don’t call foo again) and return the result of the other call to foo_coalesced. In which circumstances will your mechanism break? How can you avoid this breakage?

Processes

  • Design a mechanism for inter-process communication. Is message ordering guaranteed?

  • Design a mechanism for broadcasting messages from one process to a group of processes. Is message sordering guaranteed?

  • Consider a parent process and several children processes. Each of the children has a main loop and may be idle or processing an event/message sent either by the parent or by the OS. We want to design our system for resiliency, so we need to take into account the possibility that a child may be buggy and may freeze while handling an event/message. Design a mechanism that will let the parent determine when one child is frozen, so that it can be killed and relaunched. What are the tradeoffs you made while designing this mechanism?
